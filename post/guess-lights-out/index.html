Solve lights Out Game with Exhausting Method

@Can Yellow · Nov 23, 2018 · 1 min read

这一次，我记录下暴力枚举法求解Lights Out游戏的程序的编写思路。具体的源程序github地址

原理

对于Lights out游戏，根据给定的规则，实际上如果给定第一行的点击方案，为了保证全灭的要求，第2～n行的点击方案根据这个规则实际是唯一确定的。怎么理解呢？用文字描述就是，如果第一行任意选取一种点击方案（这里记为$S_1$）,那么为了保证全局全灭，首先要保证第一行全灭。对于第一行的状态，能够影响的只有$S_1$以及接下来操作的第二行点击方案（这里记为$S_2$，后面第i行点击方案直接记为$S_i$），也就是对应于$S_1$，$S_2$实际上是唯一的。同样考虑第二行，$S_1$根据规则会对第二行的状态产生影响，同时$S_2$本身会操作第二行，两者的效果叠加，就是第二行结束的状态。依此类推，可得倒数第二行的状态，可以确定最后一行的点击方案与最终状态，若最后一行全灭，说明上面的S组合是一组解方案，否则不是。

最后，对于第一行的可能方案，由于n个空位，都可以是0或者1，方案数量为$2^n$。

思路

根据上面的原理，可以归纳出下面的伪代码：

  1. 猜测一个首行的点击方案$S_1$;
  2. 根据点击方案确认第一行的状态$C_1$，第二行的状态$C_2$;
  3. 根据第一行的状态$C_1$，计算出第二行的点击方案$S_2$;
  4. 根据$S_2$更新上面的状态$C_2$;
  5. 根据$S_2$更新第三行的状态$C_3$;
  6. 重复上面的3，4，5步骤，直到第$n-1$行；
  7. 计算第n行的点击方案$S_n$;
  8. 更新第n行的状态$C_n$;
  9. 判断$C_n$是否满足全灭的条件，满足则解数量+1，输出点击方案$S_1,S_2,…,S_n$;否则重新猜测点击方案$S_1$回到第1步。

关键点：数据抽象与存储方式

由于每个位置的状态只有亮灭（1,0）两种可能，同时每个位置的操作也只有按与不按（1,0）两种可能，因此不论是解决方案还是状态空间每个位置都只需要一个二进制位即可。C语言中有uint64_t类型（unsigned long long int类型），在我的linux系统上uint64_t占用8个字节，即64（8×8）个二进制位。为此使用一个uint64_t类型代表一组方案或者一组状态。最大可以解决63列以内的问题（64时数据已经超了）。这里我的程序没有比较行数、列数的大小，实际上将行数列数中的较小者作为列，另外的作为行，虽然仍限制列数不能超过63列，但这样行数是可以大于63的，因为行数不受uint64_t类型的容量限制。

//解方案的存储方式，动态数组
unsigned long long int *solution=calloc(dimensionN,sizeof(unsigned long long int));

关键点：遍历初始方案$S_1$的所有可能解的方法

显然$S_1$的所有可能方案实际上是一棵二叉树的所有路径的集合，不过所学有限，我认真考虑过还是不能设计一个程序将所有的方案一一遍历。所以智能想其他方法。我们刚刚提到可以用uint64_t类型表示解决方案。更进一步的，实际上把$S$看成一个整数的话，$S_1$的所有取值是连续的从$0$到$2^n-1$.基于这一点，实际上穷举$S_1$的所有方案，就是对代表$S_1$的整数值从0遍历到$2^n-1$。

//即0xffffffffffffffff
unsigned long long int soluMax=pow(2.0,dimensionM)-1;

//soluIndex作为循环计数，同时是$S_1$的二进制表示
for(unsigned long long int soluIndex=0x0;soluIndex<=soluMax;soluIndex++)
{
    solution[0]=soluIndex;
    ...
}

关键点：方案S与C之间关系的表示——位操作

这里的位操作按照伪代码的描述涉及下面几个内容：

  1. 根据上一行的状态$C_{i-1}$，确定当前行的点击方案$S_i$;
  2. 根据当前行的点击方案$S_i$，当前行的初始状态$C_i$($C_i$的初始值是由上一行得到的，见下)，更新本行点击后的$C_i$；
  3. 确定下一行的初始状态 $C_{i+1}$，依据是上一行的点击方案$S_i$

  4. uint64_t类型高位空位的解决方案。

    uint64_t类型高位空位的解决方案

    由于实际问题是从2-63位都有可能的(1的时候，程序初值有问题，不可用)，而uint64_t是固定64位的，超过的高位都是闲置的，在计算过程中高位的值如果不处理会导致结果错误（后面会提到原因）。所以，对于每次给定的位数m，必须屏蔽（m,64]的所有高位。程序中是用与操作实现的. 令select=2^m-1,二进制下（m,64]位都是0,[0,m]位都是1，将之作为选择标记。 则target=target&select操作后，(m,64]位select为0，不论target原先为何值，这些位均置零了；[0,m]位select为1，target的这一部分与操作后会维持原样。这样就实现了屏蔽的功能，下面截取的代码中&就是起到屏蔽作用的。

    statusLast=((solution[0]<<1)&soluMax)^solution[0]^(solution[0]>>1);
...
solution[NIndex]=(~statusLast)&soluMax;
...
statusNow=((solution[NIndex]<<1)&soluMax)^solution[NIndex]^(solution[NIndex]>>1)^statusNow;

    $C_{i-1}$→$S_i$

    我们暴力求解每一步的目标是当前行i的点击必须使第i-1行全灭。即第i-1行未灭的灯，必须由点击第i对应的位置，才能灭掉。也就是 $Si$就是对$C{i-1}取反得到的。

    solution[NIndex]=(~statusLast)&soluMax;

    $S_i$→$C_i$

    因为点击第j个位置，j-1，j+1位置的灯会一起变化，只考虑j-1，其实就是对点击方案$S$做左移一位的结果，同理j+1的变化就是对方案$S$做右移一位的结果。不过对于j-1，由于高位存在的问题，左移时会将高位的默认0值发生变化，为了避免这个问题，左移后，必须人为的屏蔽高位一次，这样操作后最终$S_i$的作用效果就是三者的叠加（二进制加法也就是异或操作）。最终的状态则是把$S_i$的结果对初始$C_i$叠加一次。代码如下：

    statusNow=((solution[NIndex]<<1)&soluMax)^solution[NIndex]^(solution[NIndex]>>1)^statusNow;

    $C_{i+1}$←$S_i$

    不需要想太多，$S_i$点击后，根据规则，下一行对应位置灯的状态发生变化，两者是一致的。

    statusNext=solution[NIndex];
